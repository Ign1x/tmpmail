# TmpMail 设计思路

## 1. 项目目标

基于现有 `./DuckMail` 前端快速启动 TmpMail 项目，前端尽量复用现有页面、交互和接口契约，后端由我们使用 Rust 重新实现，先把核心邮箱主流程跑通，再逐步补齐域名管理和实时能力。

这份文档的目标不是描述所有可能性，而是明确第一阶段真正要做什么、后端需要提供哪些能力，以及按什么顺序开始实现。

## 2. 核心约束

### 2.1 前端约束

- 前端直接基于 `./DuckMail` 开发，不重做一套新的页面框架。
- 需要保留当前多账户模型、落地页转邮箱页的切换方式、域名选择器、设置面板和轮询收件逻辑。
- 第一阶段以前端已使用的 API 为准，优先兼容已有请求路径和字段结构，避免前端大改。

### 2.2 后端约束

- 后端主服务采用 Rust 实现，目标是高并发、低资源占用、清晰分层。
- 要实现实时推送，禁止轮询低效率收件。
- 域名管理必须支持公共域名和私有域名两类场景，未验证域名不能投入服务。
- 邮件接收服务与业务 API 分离部署，便于后续独立扩容。

### 2.3 兼容性约束

- 对外 API 设计优先参考 `https://vip.215.im/docs` 的风格。
- 对内落地时，必须先满足 `DuckMail` 当前已经依赖的接口：`/domains`、`/accounts`、`/token`、`/me`、`/messages`、`/messages/:id`、`/accounts/:id`。
- 前端当前通过 Next.js 代理路由转发请求，因此后端只需要稳定提供标准 HTTP API，不需要感知前端代理层细节。

## 3. 当前 DuckMail 前端基线

本文档基于仓库现状整理，关键代码位置如下：

- `DuckMail/app/[locale]/page.tsx`：首页与邮箱主页面切换、创建邮箱入口、邮件主流程。
- `DuckMail/app/api/mail/route.ts`：Next.js 代理层，转发所有邮箱相关 API。
- `DuckMail/lib/api.ts`：前端 API 契约，已明确依赖的后端接口。
- `DuckMail/contexts/auth-context.tsx`：多账户、本地持久化、账户切换与自动刷新 token。
- `DuckMail/contexts/api-provider-context.tsx`：Provider 配置、API Key 管理。
- `DuckMail/components/domain-selector.tsx`：域名选择与公共/私有域名展示逻辑。

### 3.1 已有用户流程

- 未登录时展示落地页，可一键创建邮箱或手动创建/登录。
- 支持一键创建邮箱，默认使用固定域名，注册成功后展示邮箱和密码横幅。
- 支持手动创建账户，用户输入用户名、密码、有效期并选择域名。
- 支持登录已有账户，支持完整邮箱输入或用户名加域名拆分输入。
- 支持本地保存多个账户，并在多个账户之间切换。
- 支持邮件列表、邮件详情、标记已读、删除邮件、下载原始邮件、附件展示。
- 主流程主要依赖 2 秒轮询收件，实时推送目前不是可用主路径。

### 3.2 前端已锁定的后端行为

- `GET /domains`：返回域名列表，DuckMail 前端会按 `ownerId` 和 `isVerified` 区分公共域名、私有域名和可用状态。
- `POST /accounts`：创建邮箱账户；需要允许传入 `expiresIn`。
- `POST /token`：通过邮箱地址和密码换取 JWT token。
- `GET /me`：返回当前账户信息。
- `GET /messages?page=n`：分页获取邮件列表，当前前端按每页 30 条做翻页判断。
- `GET /messages/:id`：获取邮件详情。
- `PATCH /messages/:id`：至少支持 `{ "seen": true }`。
- `DELETE /messages/:id`：删除邮件。
- `DELETE /accounts/:id`：删除账户。

设计结论：第一阶段后端必须以这组接口为最小兼容面，否则前端无法直接复用。

## 4. 总体架构设计

### 4.1 架构总览

系统分为四层：

1. `DuckMail` 前端层：保留现有 Next.js 页面和 API 代理。
2. Rust API 层：负责账户、鉴权、域名、邮件查询、管理接口。
3. 邮件接收层：负责 SMTP 接入和原始邮件落地，第一阶段明确采用独立部署的 Inbucket 作为邮件接收服务。
4. 存储与任务层：负责数据库、对象存储、域名验证任务、消息解析任务和清理任务。

推荐的职责划分如下：

- `frontend-web`：直接基于 `DuckMail` 改造，只做 UI 和代理，不承载业务状态。
- `api-server`：Rust HTTP 服务，对前端提供统一 API。
- `mail-ingest`：第一阶段采用 Inbucket 承接邮件接收与原始邮件导入，再由 Rust 服务消费和整理。
- `worker`：异步任务执行器，负责域名验证、邮件解析、清理过期数据。

### 4.2 第一阶段建议的技术实现

- Web 框架：`axum`
- 异步运行时：`tokio`
- 数据库访问：`sqlx`
- 序列化与配置：`serde`
- 日志与链路追踪：`tracing`
- 数据库：`PostgreSQL`
- 附件与原始邮件存储：S3 兼容对象存储或本地文件存储

这样做的原因很直接：Rust 主服务负责高并发 API 和任务编排，邮件接收链路可以先借助成熟组件快速起步，避免第一阶段把 SMTP、队列、解析、业务 API 一次性全写完。

## 5. Rust 后端设计细节

### 5.1 服务边界

#### A. API Server

负责：

- 账户注册、登录、删除
- JWT 签发与校验
- 域名查询与域名管理接口
- 邮件列表与详情查询
- 邮件状态更新与删除
- API Key 鉴权与私有域名授权

#### B. Domain Worker

负责：

- 生成域名验证记录
- 定时校验 DNS 记录是否生效
- 将验证成功的域名切换为可服务状态
- 可选接入 Cloudflare API，自动写入 DNS 记录

#### C. Mail Worker

负责：

- 从邮件接收层拉取或接收原始邮件
- 解析邮件头、正文、HTML、附件
- 将结构化结果写入数据库
- 将 `.eml` 和附件写入对象存储
- 更新账户未读状态和邮件索引

#### D. Cleanup Worker

负责：

- 清理过期账户
- 清理过期邮件和附件
- 清理未完成验证且超时的域名任务
- 清理失效 API Key 缓存和无用审计数据

### 5.2 核心数据模型

建议至少落以下几张核心表：

- `domains`：域名、归属用户、是否验证通过、MX/CNAME/TXT 目标值、服务状态。
- `domain_verification_tasks`：验证任务、最近一次检查结果、错误原因、下次重试时间。
- `api_keys`：API Key、归属用户、状态、权限范围。
- `accounts`：邮箱地址、密码摘要、过期时间、所属域名、状态。
- `sessions` 或 `account_tokens`：JWT 黑名单或刷新控制信息；如果第一阶段只做短期 JWT，可最小化实现。
- `messages`：发件人、主题、正文摘要、已读状态、大小、原始邮件地址、接收时间。
- `message_bodies`：纯文本正文、HTML 正文、解析结果。
- `attachments`：附件名、大小、内容类型、存储路径、所属邮件。
- `audit_logs`：账户删除、域名验证、API Key 使用等审计事件。

设计结论：账户、域名、邮件三类模型必须解耦，避免把“一个邮箱就是一个临时会话”写死在数据库里，因为前端已经明确支持多账户切换。

### 5.3 鉴权与权限模型

- 用户创建邮箱后，通过 `POST /token` 获取 JWT。
- JWT 仅用于该邮箱账户的邮件相关读写操作。
- `API Key` 用于域名查询增强和私有域名下创建账户。
- 无 API Key 时，只返回公共域名并允许在公共域名下注册。
- 有 API Key 时，除公共域名外，还返回该 Key 绑定的私有域名。
- 删除账户、读取邮件、标记已读、删除邮件都必须校验邮箱 token。
- 域名管理接口必须同时校验平台用户身份或管理型 API Key，不能复用邮箱 token。

### 5.4 API 设计原则

第一阶段对前端暴露的 API 以兼容为主，建议实现下列最小集合：

| 方法 | 路径 | 说明 |
| --- | --- | --- |
| `GET` | `/domains` | 返回可用域名列表，支持公共域名和私有域名 |
| `POST` | `/accounts` | 创建邮箱账户，支持 `expiresIn` |
| `POST` | `/token` | 登录并返回 JWT |
| `GET` | `/me` | 获取当前账户信息 |
| `GET` | `/messages?page=1` | 获取邮件列表 |
| `GET` | `/messages/:id` | 获取邮件详情 |
| `PATCH` | `/messages/:id` | 更新已读状态 |
| `DELETE` | `/messages/:id` | 删除邮件 |
| `DELETE` | `/accounts/:id` | 删除账户 |

第二阶段再补：

- 域名添加接口
- DNS 配置方案查询接口
- 手动触发域名验证接口
- 实时订阅或 Webhook/SSE 接口

第三阶段：
- 等价实现 https://vip.215.im/docs 的 API，方便用户迁移
### 5.5 邮件接收与处理链路

第一阶段建议采用“Inbucket 独立接收邮件，Rust 服务负责业务整理”的方式：

1. 用户在 Rust API 中创建邮箱账户。
2. 账户和域名映射关系写入数据库。
3. 域名的 MX 指向邮件接收服务。
4. 原始邮件进入邮件接收服务后，由 Mail Worker 拉取或订阅新邮件事件。
5. Mail Worker 解析原始邮件，抽取发件人、主题、正文、附件信息。
6. 结构化结果写入 `messages`、`message_bodies`、`attachments`。
7. 前端通过轮询 `GET /messages` 获得最新结果。

这样可以确保第一阶段重点解决“能稳定收、能稳定查、能稳定删”，而不是先陷入复杂的邮件基础设施开发。

### 5.6 附件与原始邮件策略

- 原始邮件 `.eml` 建议单独存储，`messages.downloadUrl` 指向受控下载地址。
- 附件元数据入库，二进制内容放对象存储。
- `GET /messages/:id` 返回附件数组和下载地址，满足现有前端展示。
- 删除邮件时同步执行数据库软删除和对象存储异步清理。

### 5.7 域名管理设计

设置页中的域名管理面板提供三类能力：

#### A. 添加域名

- 用户可以添加二级或三级域名。
- 后端为每个域名生成标准 DNS 配置方案。
- 域名初始状态为 `pending_verification`，未验证前不可创建邮箱。

#### B. DNS 配置方案

添加域名后，后端需要返回可直接展示给用户的 DNS 记录：

- `CNAME`：邮件子域名指向邮件服务入口
- `MX`：根域名或目标域名指向邮件服务入口
- `TXT`：域名所有权验证记录

如果后续接入 Cloudflare Key，则允许用户在设置页触发自动写入；如果不接入，保留手动配置路径即可。

#### C. 验证机制

- 支持用户手动点击验证。
- 支持定时任务自动验证。
- 验证成功后才把域名切为 `active`。
- 验证失败时记录失败原因，便于前端展示。

### 5.8 观测性与稳定性

- 所有请求打结构化日志，包含请求 ID、账户 ID、域名 ID、消息 ID。
- API 层记录 4xx/5xx 统计、接口耗时和数据库耗时。
- Worker 记录邮件解析成功率、域名验证成功率、清理任务耗时。
- 第一阶段至少要有健康检查、就绪检查和基础指标输出。

### 5.9 分阶段落地顺序

#### P0：前端兼容上线

- 跑通 `DuckMail` 前端主流程。
- 实现账户、登录、域名查询、邮件列表、邮件详情、标记已读、删除邮件。
- 用轮询作为唯一收件更新机制。

#### P1：域名管理上线

- 实现域名添加、DNS 方案生成、手动验证、自动验证。
- 打通 API Key 与私有域名权限。

#### P2：增强能力

- 实现实时消息推送。
- 引入自动 DNS 托管。
- 优化邮件解析、垃圾邮件过滤与审计能力。

## 6. 验收标准

以下标准全部满足后，第一阶段可视为可开工且目标清晰：

1. `DuckMail` 前端不需要重写主流程，只需做少量配置调整即可接新后端。
2. 后端 API 最小兼容集已经明确，能直接支撑账户、域名、邮件主流程。
3. 后端服务边界已经明确区分 API、邮件接收、异步任务三块职责。
4. 域名管理流程已经明确包含添加、DNS 展示、验证、启用四个状态步骤。
5. 第一阶段以轮询为准，实时能力被放到后续阶段，不阻塞开工。
6. 已经输出可执行的 `workflow.csv`，可以按顺序开始实现。

## 7. 本次实现结论

- 前端：基于 `./DuckMail` 开发。
- 后端：采用 Rust 实现业务 API 与任务系统。
- 邮件接收：第一阶段独立部署，可先复用成熟组件，再由 Rust 服务完成业务整合。
- 项目启动方式：先按 `workflow.csv` 顺序推进，实现最小闭环，再逐步增强。
